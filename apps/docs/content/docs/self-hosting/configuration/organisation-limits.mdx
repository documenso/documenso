On a self-hosted instance, per-organisation limits — how many documents, emails, and API
requests an organisation may use, along with feature toggles and team/member caps — are
controlled by **subscription claims** from the **admin panel**. You do not edit environment
variables or the database to change them.

This page covers three different kinds of limit. Keep them distinct:

| Limit type              | What it caps                                          | Set from the admin panel? |
| ----------------------- | ----------------------------------------------------- | ------------------------- |
| Resource quota          | Documents, emails, and API requests **per month**     | Yes — per claim and per org |
| Resource rate limit     | The same resources over a short window (e.g. `1h`)    | Yes — per claim and per org |
| Global HTTP rate limit  | API requests per IP address (100/min, hardcoded)      | No — see [Limitations](#limitations) |

## Prerequisites

- A running self-hosted Documenso instance.
- An account with the **`ADMIN`** role. This is an account-level role, separate from
  organisation and team roles. An existing admin grants it under **Admin Panel > Users >
  _(select a user)_ > Roles**, then **Update user**. The first admin on a new instance is
  created during initial setup.

Open the admin panel at `/admin`. The left sidebar contains the sections used below:
**Claims**, **Organisations**, and **Organisation Stats**.

## Viewing current usage

There are two places to see usage against limits.

### One organisation

Go to **Admin Panel > Organisations**, search for the organisation, and open it. The
**Organisation usage** section shows the current period's document, email, and API usage
against that organisation's configured quotas.

### All organisations

Go to **Admin Panel > Organisation Stats** to sort and filter monthly usage across every
organisation. You can:

- Filter by **claim** and by **period** — each period is a UTC calendar month, shown as
  `YYYY-MM`.
- Switch the view between **Show usage**, **Show usage with quotas**, and **Show daily
  averages for documents, emails and API usages**.

<Callout type="warn">
  Usage counters record **attempts**, not only successful actions. A request that would
  exceed a quota is still counted before it is rejected, so displayed usage can read higher
  than the number of actions that succeeded. The admin panel states this directly: *"these
  values may go over the actual quota, get rejected, and will still be recorded."*
</Callout>

## Subscription claims

A **subscription claim** is a named bundle of limits and feature flags (for example `Free`,
`Team`, or `Enterprise`). Claims are **templates**. When an organisation is created,
Documenso copies the relevant claim into a private, per-organisation copy. From then on, the
organisation reads from its own copy — not the template.

<Callout type="warn">
  Editing a claim template does **not** re-apply to organisations that already exist — they
  keep their own copy. To change limits for an organisation that is already running, edit
  that organisation directly (see [Change limits for one
  organisation](#change-limits-for-one-organisation)). Template edits mainly affect
  organisations created afterwards.
</Callout>

### Claim fields

Under **Admin Panel > Claims** (`/admin/claims`), each claim has these fields:

| Field                   | Controls                                                              |
| ----------------------- | -------------------------------------------------------------------- |
| **Name**                | The claim's display name.                                            |
| **Team Count**          | Teams allowed. `0` = unlimited.                                      |
| **Member Count**        | Members allowed. `0` = unlimited.                                    |
| **Envelope Item Count** | Uploaded files allowed per envelope. Minimum `1`.                    |
| **Recipient Count**     | Recipients allowed per document. `0` = unlimited.                    |
| **Feature Flags**       | Feature toggles (see [Feature flags](#feature-flags)).              |
| **Limits**              | Monthly quota and rate-limit windows for Documents, Emails, and API. |
| **Email transport**     | Email transport the claim uses. *Default (system mailer)* uses the instance default. |

#### Quotas and rate limits

The **Limits** section has three columns — **Documents**, **Emails**, and **API** — each
with two controls:

- **Monthly quota** — how many of that resource are allowed per calendar month.
- **Rate limit windows** — optional short-window caps, each a duration plus a maximum. A
  window is a number and a unit (`s`, `m`, `h`, `d`) — for example `5m`, `1h`, or `24h`.
  Each window in a resource must be unique.

<Callout type="warn">
  The **Monthly quota** fields use the **opposite** "unlimited" convention to the count
  fields above. For quotas, an **empty** field means unlimited and **`0`** blocks the
  resource entirely. For **Team**, **Member**, and **Recipient Count**, `0` means unlimited.
  The form notes this inline: *"Empty quota means unlimited, 0 blocks the resource."*
</Callout>

#### Feature flags

The **Feature Flags** section toggles capabilities for the claim, including Unlimited
documents, Branding, Hide Documenso branding, Email domains, Embed authoring, Embed signing,
White label for embed authoring/signing, 21 CFR, HIPAA, Authentication portal, Allow Legacy
Envelopes, Signing reminders, QES signing, and Disable emails.

Some flags are Enterprise features. If your license does not include a feature, its flag is
marked and cannot be newly enabled — you can still turn one off. See
[Enterprise Edition](/docs/policies/enterprise-edition) for what a license covers.

### Create or edit a claim template

{/* prettier-ignore */}
<Steps>
  <Step>
    ### Open the claims page

    Go to **Admin Panel > Claims**. The page lists every subscription claim and lets you
    search by claim ID or name.

  </Step>
  <Step>
    ### Create or select a claim

    Choose **New claim** to create one, or select an existing claim to edit it.

  </Step>
  <Step>
    ### Set the limits and flags

    Fill in the counts, **Feature Flags**, and the **Limits** section (monthly quotas and
    rate-limit windows). Leave a **Monthly quota** empty for unlimited, or set it to `0` to
    block that resource.

  </Step>
  <Step>
    ### Save

    Save the claim. Remember that template changes apply to **new** organisations, not
    existing ones.

  </Step>
</Steps>

### Change limits for one organisation

This is the reliable way to change limits for an organisation that already exists.

{/* prettier-ignore */}
<Steps>
  <Step>
    ### Open the organisation

    Go to **Admin Panel > Organisations**, search for the organisation, and open it.

  </Step>
  <Step>
    ### Edit its limits

    In the organisation's claim/limit fields, adjust the quotas, rate-limit windows, feature
    flags, or email transport. These apply directly to that organisation's own copy. The
    organisation also shows the **Inherited subscription claim** it was created from.

  </Step>
  <Step>
    ### Save

    Save your changes. They take effect for that organisation immediately.

  </Step>
</Steps>

## Usage reset

Monthly quota usage is keyed to the current **UTC calendar month**. There is no scheduled
reset job — when the month rolls over, the counter for the new period starts at `0`. So an
organisation's monthly document, email, and API usage resets at the start of each UTC month.

## Limitations

- **The global HTTP rate limit is not configurable.** Documenso enforces a hardcoded limit
  of **100 requests per minute per IP address** on its API endpoint groups (`/api/v1`,
  `/api/v2`, and the tRPC API are each limited separately). Exceeding it returns `429 Too
  Many Requests`. This is a per-IP anti-abuse safeguard applied at the HTTP layer — it is
  **not** per-organisation, is not stored on any claim, and cannot be changed from the admin
  panel. See [Rate Limits](/docs/developers/api/rate-limits).
- **Claim template edits are not retroactive.** They do not re-apply to organisations that
  already exist. Edit the organisation directly instead.

## Troubleshooting

| Symptom                                        | Cause and fix                                                                                                                                              |
| ---------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| An organisation hit its limit unexpectedly     | Usage counts **attempts**, including rejected over-quota requests. Compare recorded usage against the quota under **Organisation Stats > Show usage with quotas**. |
| A resource is blocked entirely, not just capped | The **Monthly quota** is set to `0`, which blocks the resource. Leave it **empty** for unlimited.                                                          |
| Emails are not sending for an organisation      | Besides the email quota, check whether the **Disable emails** feature flag is enabled on the organisation's claim — it blocks all emails regardless of quota. |
| A claim template edit did not take effect       | Template edits are not retroactive. Edit the organisation directly under **Admin Panel > Organisations**.                                                   |
